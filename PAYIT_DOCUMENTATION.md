# PayIT - DocumentationTelegram bot for USDC on Arc (Circle's stablecoin L1), with Naira off-rampvia Paj Cash, SMS verification via Termii, and informational stablecoinyield lookup. Non-custodial: every user holds their own wallet.**Status: testnet skeleton.** Working starting point, not a finished,audited production system. See "What's not built yet" before real money.\---## 1\. Architecture: non-custodialEach user gets their own independently-generated Arc wallet(`ethers.Wallet.createRandom()`) at onboarding - **not** derived from anyshared seed. That's deliberate: an earlier version derived every user fromone master mnemonic, which meant a single leaked child key could,depending on derivation hardening, expose other users' keys. Independentwallets have no relationship to each other - exporting one reveals nothingabout anyone else's.The private key is encrypted with AES-256-GCM, using a key derived from theuser's own 4-digit PIN via `scrypt`. PayIT's server never stores a usablecopy of anyone's private key - only the correct PIN can decrypt it, anddecryption happens transiently in memory, just long enough to sign atransaction or show the user their own key on request.**This means: if a user forgets their PIN and never separately backed uptheir key, those funds are permanently unrecoverable.** There is no backendrecovery path - that's what non-custodial actually means, not a bug. Makesure this is communicated clearly to users (the bot does this atonboarding).**Telegram security note:** Telegram bot messages are not end-to-endencrypted. Showing a raw private key in chat (even briefly, even withauto-delete) has some inherent exposure. The bot auto-deletes key-exportmessages after 60 seconds as a mitigation, but a user screenshotting it orTelegram itself logging it server-side are both outside the bot's control.\---## 2\. The ARC-cli / Canteen tool`the-canteen-dev/ARC-cli` (installed via `uv tool install git+https://...`)was inspected before recommending it: it's a legitimate developer tool fromCanteen, who run official Circle/Arc-partnered hackathons. It provides anRPC key for a Canteen-hosted Arc testnet node, posts hackathon progressupdates, and bundles Arc/Circle docs + sample codebases for coding-agentcontext (`arc-canteen context sync`, pulling `the-canteen-dev/context-arc`into `\\\\\\\~/.arc-canteen/context/`). The source was read line-by-line - nowallet, key, or mnemonic handling anywhere in it.**Use it for:** getting verified Arc sample code/addresses (e.g. for theSwap feature below) instead of guessing. After `context sync`, point acoding agent at `\\\\\\\~/.arc-canteen/context/`.If `arc-canteen` isn't found after install, run `uv tool update-shell` andopen a **new** terminal window (PATH changes don't apply to the one youran the install in).\---## 4\. Why there's no "export private key" caveat anymoreThe earlier custodial design deliberately left this out, because in acustodial model there's no private key meaningfully owned by the user. Nowthat the architecture is genuinely non-custodial, `/export` is the correct,safe version of that feature: each user's key really is their own, exportingit reveals nothing about any other user, and the server can't silentlyre-derive it the way it could under the old shared-seed design.\---## 6\. Swap - not live yet, by designCircle's official Arc ecosystem partners include Curve, Uniswap Labs,Aerodrome/Velodrome, Euler, and Fluid as testnet participants, so real swapvenues exist. `src/swap.js` has working, generic Uniswap-V2-style swaplogic (quote + approve + swap, with mandatory slippage protection) - but`SWAP\\\\\\\_ROUTER\\\\\\\_ADDRESS` is intentionally left blank in `.env.example`,because hardcoding a guessed contract address risks sending real (test)funds to the wrong place with no way back.To turn this on: get a **verified** router address (via `arc-canteen context sync`, or directly from whichever DEX's own testnet docs you pick),set `SWAP\\\\\\\_ROUTER\\\\\\\_ADDRESS` in `.env`, and confirm its function signaturesmatch the ABI in `src/swap.js` before trusting it with real transactions."Swap to Naira" isn't a swap - Naira isn't a token on Arc - that's just`/withdraw`.\---## 7\. Savings/Yield - informational only, intentionally not autonomous`/yields` (or the \\u{1F4B0} Yields button) surfaces real current stablecoinAPYs from DeFiLlama's free public API, for reference. It does **not** moveany money, on Arc or anywhere else, and there's no path in this codebasefor it to do so.This was a deliberate choice, not a missing feature: "an agent autonomouslysearches for and allocates funds to the best APY" is exactly the patternthat causes real losses in DeFi (rug pulls, unaudited contracts, incentive-APYs that collapse once token rewards end). Arc itself is testnet-only withno established trust record on which pools are safe. If you build thisfurther, keep a human explicitly confirming every individual deposit -never let the bot move funds into a yield position on its own.\---## 8\. SMS OTP (Termii)`src/otp.js` implements Termii's documented Send/Verify Token API(`POST /api/sms/otp/send` and `/api/sms/otp/verify`). This is built fromtheir published docs, not guessed - but test it against your real account,since sender ID approval and routing can vary by account configuration.\---## 9\. What you still need to provide1. **Termii API key + sender ID** - sign up at termii.com2. **Paj Cash real API docs + credentials**, including the actual`PAJCASH\\\\\\\_OFFRAMP\\\\\\\_ADDRESS` to send USDC to - `src/offramp.js` is still aplausible-shaped placeholder for the notification call3. **A verified Arc DEX router address** if you want Swap live4. **Card-issuing partner decision** - nothing exists for card spend yet5. **KYC approach** - whose flow gates withdrawals/phone verification, andat what size6. **Regulatory structure confirmation** - whether PayIT operates underPaj Cash's CBN license or needs its own, given it now holds no custodyof funds at all (this may actually simplify the regulatory picturecompared to the earlier custodial design - worth confirming with counsel)7. **Hosting** for 24/7 uptime once this is more than a local test\---## 10\. What's not built yet* Card spending* Real swap execution (blocked on a verified router address)* Actual yield deposits (intentionally - see Section 7)* Bills, Bulk Send, Schedule* KYC/AML gating* PIN attempt rate-limiting (no lockout after repeated wrong PINs yet)* Treasury/gas-sponsorship - users now pay their own gas from their ownwallet; there's no mechanism yet to top up a user who's run out of gasbut still has USDC they want to move (a real UX gap worth solving later)* Double-entry accounting / audit trail beyond the simple transaction log* Rate limiting, abuse protection, error monitoring, backups\---## 11\.
